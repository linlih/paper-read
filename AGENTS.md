# AGENTS.md

本文件适用于仓库根目录下的全部代码和文档。后续代理或开发者在修改本项目时，必须优先遵守这里的约定；若更深层目录以后新增自己的 `AGENTS.md`，以更深层文件为准。

## 项目定位

Paper Commons 是一个论文阅读与标注原型，当前目标是把“导入论文、阅读 PDF/文本、保存笔记和讨论、形成可共享的本地数据”做扎实。

- 优先服务阅读、标注、笔记、讨论这些核心工作流。
- 不要把项目改成营销页、通用文献管理器或重型后端系统。
- 新能力应参考 `TODO.md` 的优先级推进，避免越过当前 MVP 边界做大改。
- AI 相关输出必须最终能够追溯到原文页码、段落、坐标或引用位置；在没有真实依据链路前，不要把启发式摘要包装成可靠结论。

## 技术栈约束

当前项目刻意保持轻量：

- 主前端：React + Vite + Tailwind/shadcn-style components，来源为 `frontend/`。
- 旧原生 `index.html` / `app.js` / `styles.css` 仅作为 legacy 原型保留，迁移完成后再删除。
- 新阅读器仍必须保持 HTML-first、工具型、紧凑、可扫描，不做营销页。
- 后端：Node.js 内置模块 `http`、`fs`、`path`，不依赖框架。
- 持久化：浏览器 `localStorage` + 本地 `data/store.json` 共享存储。
- PDF 阅读：优先使用页面中引入的 PDF.js CDN，缺失时保留兼容降级。

修改时遵守：

- 不要引入构建工具、前端框架、CSS 框架或包管理器，除非任务明确要求且收益充分。
- 不要把当前单页原型拆成复杂目录结构，除非功能已经明显超过单文件可维护边界。
- 不要提交密钥、Token、用户隐私数据或大型 PDF 文件。
- `data/store.json` 是本地运行数据，修改数据结构时必须兼容旧数据或提供清晰迁移路径。

## 代码规范

- 使用 ASCII 标点和代码字符；中文界面文案可以使用中文。
- JavaScript 使用现有风格：`const` / `let`、双引号、分号、函数声明，避免无必要的抽象层。
- 渲染用户可控内容时必须使用 `escapeHtml` 或等价转义，禁止把未转义输入直接拼进 `innerHTML`。
- 所有网络、文件、JSON、PDF 加载路径都要有失败兜底；本地静态打开和 `node server.js` 运行都应尽量可用。
- 新增状态字段时，同步考虑 `defaultStore`、`loadStore`、`saveStore`、服务端存储和已有本地数据。
- 服务器静态文件路径必须继续限制在项目根目录内，不能放宽 `resolveStaticPath` 的路径逃逸保护。
- API 返回 JSON 时保持 `{ ok: true }`、`{ error: "..." }` 或 `{ paper/store: ... }` 这类简单结构，不引入复杂协议。

## 前端与交互规范

- 保持工具型、紧凑、可扫描的阅读工作台风格，不做营销式 hero、装饰性背景或大面积视觉噪音。
- UI 文案使用简洁中文，直接服务用户动作；不要在界面里堆说明性教程。
- 保持现有视觉系统：CSS 变量、6px 圆角、浅色界面、克制的蓝色强调色。
- 主要交互必须具备清晰的空状态、加载失败状态和不可用状态。
- 新增按钮、输入、标签页或工具条时，保证移动端和窄屏不溢出、不遮挡、不造成布局跳动。
- PDF 标注、文本选区、笔记和讨论的锚点必须稳定，至少保留页码或章节名。

## 文档规范

- 调研文档必须写明更新时间、信息来源和不确定性。
- 涉及价格、模型、API、产品能力等可能变化的信息，更新前必须重新核验，不能只凭旧记忆。
- `TODO.md` 只记录产品执行顺序和状态，不写长篇实现细节。
- 新增技术方案文档时，优先给出结论、适用场景、风险和下一步验证方式。

## 验证方式

本项目目前没有自动化测试脚本。改动后至少执行适用的轻量检查：

- JavaScript 语法检查：`node --check app.js` 和 `node --check server.js`
- 本地服务启动：`node server.js`
- 健康检查：`curl http://localhost:4000/api/health`
- 前端冒烟：打开 `http://localhost:4000/`，确认论文列表、阅读模式切换、笔记/讨论保存、PDF 降级或渲染路径正常。

如果端口被占用，使用 `PORT=其他端口 node server.js`。完成任务时说明实际执行过哪些检查，以及没有执行的原因。

## 变更纪律

- 保持改动范围小，优先修正用户请求涉及的行为。
- 不要顺手重写无关 UI、数据、调研文档或存储内容。
- 如果发现已有本地改动，默认它们属于用户，不要回滚。
- 修改共享数据结构、路由、存储文件或解析流程前，先确认旧入口仍能工作。
- 对外部服务调用要设置合理错误处理，避免一个 API 失败导致整个阅读器不可用。
